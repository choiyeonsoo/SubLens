1. 기술 스택

구분 기술
Backend Spring Boot 3.x, Spring Security, JPA (Hibernate), Java 21
인증 JWT (jjwt 0.11.5), HttpOnly Cookie, Refresh Token Rotation
DB PostgreSQL 16
캐시/세션 Redis 7 (Lettuce)
메일 Resend API (REST)
Frontend Next.js 16, React 19, TanStack Query, Axios, Tailwind CSS
인프라 Docker Compose (postgres, redis, backend, frontend, nginx)

2. 아키텍처 개요

[Frontend]  →  Axios (withCredentials)  →  [Backend :8080]

    ↕ HttpOnly Cookie (access_token, refresh_token)

[Backend]  →  JPA → [PostgreSQL]

[Backend]  →  Lettuce → [Redis] (refresh token, reset token 저장)

[Backend]  →  REST → [Resend API] (비밀번호 재설정 이메일)

  • Stateless: 세션 미사용, JWT만으로 인증
  • CORS: http://localhost:* 패턴 허용, allowCredentials=true
  • CSRF: 비활성화 (JWT + Cookie 기반이므로)

3. 데이터 모델

3.1 User 엔티티 (users 테이블)

필드 타입 제약조건 설명
id Long PK, AUTO_INCREMENT 사용자 고유 ID
email String UNIQUE, NOT NULL 이메일 (로그인 ID)
password String NOT NULL BCrypt 해시 비밀번호
name String NOT NULL 이름
phoneNumber String nullable 전화번호
role Enum (USER, ADMIN) 권한

3.2 DTO

클래스 필드 용도
SignupRequest email, password, name, phoneNumber 회원가입 요청
LoginRequest email, password 로그인 요청
LoginResponse accessToken, refreshToken 토큰 응답 (내부용)
PasswordResetRequest email 비밀번호 재설정 요청
PasswordReset token, password 비밀번호 재설정 실행

4. API 명세

Base Path: /api/auth  |  모든 인증 관련 API는 permitAll()

4.1 회원가입

항목 내용
Endpoint POST /api/auth/signup
Request Body { "email", "password", "name", "phoneNumber" }
성공 응답 200 "회원가입 완료"
실패 409 DUPLICATE_EMAIL

1. 이메일 중복 검사 → 2. BCrypt 해시 → 3. User 저장 (Role=USER)

4.2 로그인

항목 내용
Endpoint POST /api/auth/login
Request Body { "email", "password" }
성공 응답 200 (Body 없음, 쿠키에 토큰 세팅)
실패 404 EMAIL_NOT_FOUND, 401 INVALID_PASSWORD

1. 이메일로 사용자 조회 → 2. 비밀번호 검증 (BCrypt)

3. Access Token + Refresh Token 생성 → 4. Redis에 Refresh Token 저장 (24h)

5. HttpOnly Cookie 2개 세팅 (access_token: 30분, refresh_token: 24시간)

4.3 내 정보 조회

항목 내용
Endpoint GET /api/auth/me
인증 필요 (Cookie의 access_token)
성공 응답 200 { "userId", "roles" }
실패 401

4.4 로그아웃

항목 내용
Endpoint POST /api/auth/logout
성공 응답 200

1. Redis에서 refresh:{userId} 삭제

2. access_token, refresh_token 쿠키 MaxAge=0 으로 삭제

4.5 토큰 재발급 (Reissue)

항목 내용
Endpoint POST /api/auth/reissue
인증 Cookie의 refresh_token 필요
성공 응답 200 (새 토큰 쿠키 세팅)
실패 401 TOKEN_EXPIRED

1. 쿠키에서 Refresh Token 추출 → 2. JWT 유효성 검사

3. Redis의 저장된 값과 비교 → 4. 새 Access + Refresh 발급

5. Redis 값 교체 (Rotation) → 6. 새 쿠키 세팅

4.6 비밀번호 재설정 요청 (토큰 발행 + 이메일)

항목 내용
Endpoint POST /api/auth/request-reset
Request Body { "email" }
응답 항상 200 (보안상 이메일 존재 여부 노출 안 함)

1. 이메일로 사용자 조회 (없으면 조용히 return)

2. 기존 토큰이 있으면 무효화: reset:user:{userId} 조회 → reset:token:{기존토큰} 삭제

3. 새 UUID 토큰 생성

4. 듀얼 키 Redis 저장 (TTL 10분):

    reset:user:{userId} → newToken

    reset:token:{newToken} → userId

5. Resend API로 재설정 이메일 발송 (링크: /reset-password?token=xxx)

4.7 재설정 토큰 검증

항목 내용
Endpoint GET /api/auth/validate-token?token={token}
성공 응답 200
실패 400 INVALID_RESET_TOKEN

Redis에서 reset:token:{token} 조회 → 없으면 예외

4.8 비밀번호 재설정 실행

항목 내용
Endpoint POST /api/auth/reset-password
Request Body { "token", "password" }
성공 응답 200
실패 400 INVALID_RESET_TOKEN, 404 EMAIL_NOT_FOUND

1. reset:token:{token}에서 userId 조회 → 2. 사용자 조회

3. 새 비밀번호 BCrypt 해시 후 저장 → 4. Redis 토큰 삭제 (일회성 보장)

5. JWT 토큰 구조

5.1 Access Token

Claim 값
sub userId (Long → String)
role "USER" 또는 "ADMIN"
iat 발급 시간
exp 발급 + access-token-expiration

5.2 Refresh Token

Claim 값
sub userId
iat 발급 시간
exp 발급 + refresh-token-expiration

서명: HMAC-SHA256 (jwt.secret 기반)

5.3 Cookie 설정

쿠키명 HttpOnly Secure Path MaxAge
access_token true false (개발) / 30분
refresh_token true false (개발) / 24시간

6. Security Filter 흐름

요청 수신
  │
  ├─ shouldNotFilter 체크 (/login, /signup, /reissue → 필터 Skip)
  │
  ├─ 쿠키에서 access_token 추출
  │    ├─ null → 그대로 필터 통과 (미인증 상태)
  │    └─ 존재 →
  │         ├─ validateToken() 성공 → getAuthentication() → SecurityContext 저장
  │         └─ validateToken() 실패 → SecurityContext 초기화, JSON 에러 응답 반환
  │
  └─ Controller 진입

7. Redis 키 설계

키 패턴 값 TTL 용도
refresh:{userId} Refresh Token (JWT) 24시간 로그인 세션 유지, Rotation 검증
reset:user:{userId} Reset Token (UUID) 10분 유저당 1개 보장, 기존 토큰 무효화용
reset:token:{token} userId 10분 URL 토큰 → 유저 역방향 조회

듀얼 키 전략: 새 재설정 요청 시 reset:user:{userId}에서 기존 토큰을 찾아 reset:token:{기존토큰}을 삭제 후, 새 토큰 쌍을 저장합니다. 이로써 유저당 유효한 재설정 토큰은 항상 1개만 존재합니다.

8. 에러 코드 정의

코드 HTTP Status 메시지
INTERNAL_SERVER_ERROR 500 서버 내부 오류
INVALID_REQUEST 400 잘못된 요청
TOKEN_EXPIRED 401 토큰이 만료되었습니다.
INVALID_SIGNATURE 401 토큰 서명이 유효하지 않습니다.
MALFORMED_TOKEN 400 토큰 형식이 올바르지 않습니다.
UNSUPPORTED_TOKEN 400 지원되지 않는 토큰입니다.
EMPTY_TOKEN 400 토큰이 존재하지 않습니다.
EMAIL_NOT_FOUND 404 존재하지 않는 이메일입니다.
INVALID_PASSWORD 401 비밀번호가 일치하지 않습니다.
DUPLICATE_EMAIL 409 이미 존재하는 이메일입니다.
INVALID_RESET_TOKEN 400 유효하지 않거나 만료된 재설정 토큰입니다.

에러 응답 형식

{
  "success": false,
  "code": "TOKEN_EXPIRED",
  "message": "토큰이 만료되었습니다.",
  "data": null
}

10. 프론트엔드 API 연동

10.1 Axios 인스턴스

const api = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_BASE_URL,
  withCredentials: true,      // HttpOnly Cookie 전송 핵심
  headers: { "Content-Type": "application/json" },
});

10.2 API 함수 (features/auth/api.ts)

함수 HTTP 엔드포인트
login(email, password) POST /api/auth/login
signup(email, password, name, phoneNumber) POST /api/auth/signup
requestPasswordReset(email) POST /api/auth/request-reset
validateToken(token) GET /api/auth/validate-token?token=
resetPassword(token, password) POST /api/auth/reset-password

10.3 React Query Hooks (features/auth/hooks.ts)

Hook 종류 용도
useLogin() useMutation 로그인
useSignup() useMutation 회원가입
useRequestPasswordReset() useMutation 재설정 이메일 요청
useValidateToken(token) useQuery 토큰 검증 (자동 실행)
useResetPassword() useMutation 비밀번호 변경 실행

11. 설정값 (application.yaml)

키 현재값 비고
jwt.access-token-expiration 10,000ms (10초) 개발용. 운영: 1,800,000ms (30분)
jwt.refresh-token-expiration 600,000ms (10분) 개발용. 운영: 86,400,000ms (24시간)
spring.data.redis.host localhost Docker: redis
spring.data.redis.port 6379
resend.api.key 환경변수 Resend 메일 API 키
