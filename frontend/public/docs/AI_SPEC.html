2. 전체 AI 아키텍처

사용자 자연어 질문

    ↓

Spring Boot (기존) → POST /api/ai/query → Python AI 서비스 (신규 Docker 컨테이너)

    ↓

[1단계] 분류기 — Claude Sonnet으로 질문 유형 판단 (max_tokens: 100, 저비용)

    ↓

[2단계] LangChain Agent — 유형별 경로 실행

    ├─ 타입 1: 내 데이터 조회   → SQL 경로

    ├─ 타입 2: 서비스 안내     → RAG (SUBLENS FAQ)

    ├─ 타입 3: 번들 상품 문의  → RAG (번들 카탈로그 문서)

    └─ 타입 4: 구독 최적화 추천 → SQL + 번들 DB + Claude 추론

    ↓

[3단계] Claude Sonnet 4.x → 자연어 응답 생성 → 사용자

3. 질문 타입 4가지 상세

타입 1 — 내 데이터 조회 (SQL)

  • "이번 달 구독료 얼마야?"
  • "올해 예상 구독 지출은?"
  • "넷플릭스 구독 시작한 게 언제야?"

→ 사용자 DB (subscriptions) SQL 조회

타입 2 — 서비스 안내 (RAG)

  • "구독 어떻게 추가해?"
  • "무료 플랜 한도가 뭐야?"
  • "결제 수단 변경 방법은?"

→ SUBLENS FAQ 문서 벡터 검색 (source = 'sublens_faq')

타입 3 — 번들 상품 문의 (RAG)

  • "LGU+ 유독 약정 조건이 뭐야?"
  • "SKT T우주 중도 해지하면?"
  • "네이버 플러스 가족 공유 돼?"

→ 번들 카탈로그 문서 벡터 검색 (source = 'bundle_detail')

타입 4 — 구독 최적화 추천 (복합)

  • "내 구독 최적화해줘"
  • "통신사 번들이 나한테 이득이야?"
  • "네이버 vs LGU+ 유독 뭐가 나아?"

→ SQL(내 구독 현황) + 번들 카탈로그 DB + Claude 추론

→ 체크박스 혜택 선택값은 DB 저장 없이 요청 파라미터로 전달

→ 약정 등 단점은 ⚠️ 고지하되 최종 선택은 사용자에게 맡김

4. 질문 유형 분류기 프롬프트 명세 NEW

타입 1~4 분류 정확도는 전체 응답 품질을 결정하는 가장 중요한 단계다. 분류 오류는 완전히 다른 경로를 실행시키므로, 아래 프롬프트와 퓨샷 예시를 그대로 사용한다.

4.1 분류기 API 호출 설정

항목 값 이유
model claude-sonnet-4-20250514 Haiku보다 분류 정확도 높음
max_tokens 100 JSON 결과만 받으므로 낮게 설정 (비용 절감)
temperature 0 분류 일관성 확보

4.2 시스템 프롬프트

You are a query classifier for SUBLENS, a subscription management platform.
Classify the user's question into exactly one of four types.

TYPE DEFINITIONS
- type_1: Queries about the user's own subscription data (amounts, dates, counts, history).
  Requires SQL lookup against the user's subscription records.
- type_2: Queries about how to use SUBLENS itself (features, limits, billing, UI).
  Answered from SUBLENS FAQ documents only.
- type_3: Queries about specific bundle product terms, conditions, or specs
  (contracts, cancellation, family sharing, eligibility).
  Answered from bundle catalog documents only.
- type_4: Queries asking for optimization, comparison, or recommendations
  involving the user's own subscriptions AND bundle products together.

AMBIGUITY RULES
- If the question could be type_1 OR type_4, choose type_4.
  (e.g. "내 구독이 번들이랑 겹쳐?" → type_4, not type_1)
- If the question mentions a bundle name but only asks about its terms, choose type_3.
  (e.g. "유독 해지하면 위약금 있어?" → type_3, not type_4)
- If completely unrelated to subscriptions, classify as type_2 and answer with FAQ fallback.

OUTPUT FORMAT — return only valid JSON, no explanation, no markdown:
{"type": "type_1"|"type_2"|"type_3"|"type_4", "confidence": 0.0-1.0, "reason": "one sentence"}

4.4 엣지 케이스 처리 규칙

패턴 분류 근거
"나한테", "내 구독이랑" + 번들 언급 type_4 1인칭 + 번들 조합은 비교 추천 의도
번들 이름 단독 + 약관/조건 질문 type_3 추천 의도 없으면 번들 RAG로 충분
confidence < 0.55 type_2 폴백 "더 자세히 말씀해주시면" 응답 트리거
구독과 무관한 질문 type_2 폴백 FAQ 폴백으로 안내

4.5 FastAPI 분류기 호출 코드

CLASSIFIER_SYSTEM_PROMPT = """...(4.2 내용)..."""
FEWSHOT_EXAMPLES = """...(4.3 내용)..."""

async def classify_query(user_question: str) -> dict:
    response = await anthropic_client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=100,
        system=CLASSIFIER_SYSTEM_PROMPT + "\n\n" + FEWSHOT_EXAMPLES,
        messages=[{"role": "user", "content": user_question}]
    )
    result = json.loads(response.content[0].text)

    # 낮은 확신 → FAQ 폴백
    if result["confidence"] < 0.55:
        result["type"] = "type_2"

    return result
    # {"type": "type_4", "confidence": 0.97, "reason": "..."}

5. SQL 경로 상세 (NL2SQL + 3단 보안 레이어)

LLM이 SQL을 자유롭게 생성하되, 아래 3단 보안으로 안전하게 통제한다.

단계 방법 목적
1. 스키마 제한 LLM에게 subscriptions, users 테이블만 노출. password_hash 등 민감 컬럼은 프롬프트에서 제외 민감 데이터 접근 차단
2. userId 강제 주입 LLM이 생성한 SQL에 서버 레벨에서 WHERE user_id = :currentUserId 조건을 실행 직전에 무조건 append 타인 데이터 조회 방지
3. SELECT 전용 계정 AI 서비스가 사용하는 PostgreSQL 계정에 INSERT/UPDATE/DELETE 권한 물리적 차단 쓰기 쿼리 실행 불가

6. RAG 경로 상세

사전 인덱싱 (문서 등록 시 1회)

원본 문서 → 500자 단위 청킹 → 각 청크 임베딩(벡터 변환) → PGVector 저장

실시간 검색 (질문마다)

사용자 질문 → 질문만 임베딩 → PGVector 코사인 유사도 검색 → 상위 3개 청크 추출

→ Claude에게 "이 문서 참고해서 답해줘" → 자연어 응답

document_chunk 테이블의 source 컬럼으로 타입 2/3 구분:

source 값 용도
sublens_faq 타입 2 — SUBLENS 서비스 안내
bundle_detail 타입 3 — 번들 상품 약정/조건 상세

7. Conversation Memory (대화 이어가기)

LangChain ConversationSummaryBufferMemory로 세션 내 대화 히스토리 유지. Redis에 session_id 기준으로 저장 (SUBLENS 기존 Redis 재사용). Redis TTL은 2시간으로 설정 — 페이지 새로고침이나 짧은 이탈 후 재방문 시에도 대화를 이어갈 수 있도록 한다.

BufferWindowMemory 대신 SummaryBufferMemory를 사용하는 이유:
BufferWindowMemory(k=8)는 8턴 초과 시 오래된 맥락을 완전히 잘라버린다. SummaryBufferMemory는 5,000 토큰 초과분을 Haiku로 요약 압축하므로 맥락이 완전히 소실되지 않으며, 요약 비용도 Haiku 기준으로 매우 저렴하다. 
from langchain.memory import ConversationSummaryBufferMemory
from langchain_anthropic import ChatAnthropic

# 요약은 저렴한 Haiku로 처리
summarizer_llm = ChatAnthropic(
    model="claude-haiku-4-5-20251001",
    max_tokens=300
)

memory = ConversationSummaryBufferMemory(
    llm=summarizer_llm,
    max_token_limit=5000,   # 이 초과분부터 요약으로 압축
    return_messages=True
)

# 세션당 Redis 키: f"conv_memory:{session_id}"
# 직렬화: memory.save_context() → Redis SET (TTL 7200초)

session_store = {}  # Redis로 관리

def get_chain(session_id: str):
    if session_id not in session_store:
        memory = ConversationSummaryBufferMemory(
            llm=summarizer_llm,
            max_token_limit=5000,
            return_messages=True
        )
        session_store[session_id] = ConversationChain(llm=claude, memory=memory)
    return session_store[session_id]

사용자가 "그러면 추천이 뭐야?"처럼 이전 맥락을 이어서 질문해도 앞 대화를 기억하고 답변한다. Redis TTL 만료(2시간) 시 메모리 초기화 (의도적 설계).

8. Conversation Memory 토큰 예산 NEW

상한 없이 ConversationBufferMemory를 사용하면 장기 세션에서 히스토리만 13,000 토큰 이상 누적되어 Claude API 비용이 예측 불가능해진다. 반드시 아래 예산 기준을 준수한다.

8.1 API 호출 1회당 입력 토큰 구성 (타입 4 기준, 최악 케이스)

구성 요소 토큰 수 비고
시스템 프롬프트 ~800 고정
RAG 청크 (상위 3개 × 500자) ~1,500 한국어 1자 ≈ 1토큰
SQL 결과 (구독 목록) ~500 구독 10건 기준
현재 사용자 질문 ~50 평균 30~70자
대화 히스토리 (max_token_limit) ~5,000 SummaryBufferMemory 상한
총 입력 (권장 상한) ~7,850

8.2 실제 대화 1턴 토큰 감각 (한국어 기준)

메시지 토큰 수 예시
짧은 사용자 질문 ~15 "내 구독 최적화해줘"
일반 사용자 질문 ~50 "넷플릭스랑 유튜브 프리미엄 합치면 번들이 나아?"
Claude 짧은 답변 ~200 2~3줄 요약 답변
Claude 분석 답변 ~650 번들 비교 + 비용 계산 포함
1턴 평균 ~700 한국어는 영어 대비 약 1.3~1.5배 토큰 소모
max_token_limit = 5,000 의 실질 의미:
1턴 평균 700 토큰이면 약 7왕복(7개 질문 + 7개 답변)까지 그대로 유지된다. 이를 초과하는 오래된 대화는 Haiku가 요약으로 압축하므로 맥락이 완전히 소실되지 않는다.

8.3 메모리 전략별 비교

전략 장점 단점 권장 여부
ConversationBufferMemory 구현 단순 무제한 누적, 비용 예측 불가 ❌ 사용 금지
BufferWindowMemory(k=8) 상한 명확 8턴 초과 시 맥락 완전 소실 △ 단순 QA에만
SummaryBufferMemory(max=5000) 맥락 보존 + 비용 통제 요약 LLM 호출 발생 ✅ 권장

8.4 Redis 저장 키 설계

키 패턴 값 TTL 용도
conv_memory:{session_id} 직렬화된 Memory 객체 (JSON) 7,200초 (2시간) 대화 히스토리 + 요약 상태 보존

10.2 bundle_catalog

CREATE TABLE bundle_catalog (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    provider        VARCHAR(50)  NOT NULL,   -- 'LGU+', 'SKT', 'KT', 'NAVER', 'COUPANG'
    provider_url    TEXT,
    plan_name       VARCHAR(200) NOT NULL,
    plan_url        TEXT,
    description     TEXT,
    includes        TEXT[]       NOT NULL,   -- {'넷플릭스','유튜브프리미엄'}
    base_price      INTEGER      NOT NULL,
    original_price  INTEGER,
    discount_rate   INTEGER,
    contract_months INTEGER,                 -- null이면 무약정
    telecom_exclusive VARCHAR(20),           -- 'SKT','KT','LG U+', null이면 누구나 가입 가능
    has_options     BOOLEAN      NOT NULL DEFAULT false,
    is_active       BOOLEAN      NOT NULL DEFAULT true,
    crawled_at      TIMESTAMP    NOT NULL DEFAULT now(),
    created_at      TIMESTAMP    NOT NULL DEFAULT now(),
    updated_at      TIMESTAMP
);

CREATE INDEX idx_bundle_catalog_provider    ON bundle_catalog(provider);
CREATE INDEX idx_bundle_catalog_is_active   ON bundle_catalog(is_active);
CREATE INDEX idx_bundle_catalog_includes    ON bundle_catalog USING GIN(includes);
-- 사용 예: SELECT * FROM bundle_catalog WHERE includes @> ARRAY['넷플릭스']

10.3 bundle_option

CREATE TABLE bundle_option (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    bundle_id    UUID NOT NULL REFERENCES bundle_catalog(id) ON DELETE CASCADE,
    option_name  VARCHAR(200) NOT NULL,   -- '넷플릭스 스탠다드'
    option_spec  VARCHAR(300),            -- '동시접속 2명, FHD 화질'
    extra_price  INTEGER NOT NULL DEFAULT 0,
    total_price  INTEGER NOT NULL,
    is_active    BOOLEAN NOT NULL DEFAULT true,
    created_at   TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX idx_bundle_option_bundle_id ON bundle_option(bundle_id);

10.4 bundle_price_history

CREATE TABLE bundle_price_history (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    bundle_id     UUID NOT NULL REFERENCES bundle_catalog(id) ON DELETE CASCADE,
    old_price     INTEGER NOT NULL,
    new_price     INTEGER NOT NULL,
    change_reason VARCHAR(100),
    changed_at    TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX idx_price_history_bundle_id  ON bundle_price_history(bundle_id);
CREATE INDEX idx_price_history_changed_at ON bundle_price_history(changed_at);

10.5 document_chunk (RAG용 PGVector)

CREATE TABLE document_chunk (
    id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source    VARCHAR(100) NOT NULL,  -- 'sublens_faq' or 'bundle_detail'
    source_id UUID,                   -- bundle_catalog.id (번들 상세면)
    content   TEXT NOT NULL,
    embedding vector(1536),
    created_at TIMESTAMP NOT NULL DEFAULT now()

CREATE INDEX idx_document_chunk_embedding
    ON document_chunk
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);

11. 크롤링 전략

11.1 사이트별 크롤링 방법 (직접 확인 완료)

사이트 URL 방법 로그인 봇차단 상태
LGU+ 유독 lguplus.com/pogg/category/전체 get_page_text + click 모달 불필요 없음 즉시 가능
SKT T우주 m.sktuniverse.co.kr/category/... get_page_text 불필요 없음 즉시 가능
KT 구독 my.kt.com/product/OttSubscribeView.do innerText + 탭 클릭 불필요 없음 즉시 가능

쿠팡 로켓와우 — 수동 입력 — — 단일 상품 구조
봇 감지 대응: 현재 "봇차단 없음"은 현 시점 기준이다. LGU+ 유독, KT는 세션·헤더 기반 차단이 추가될 수 있다. 크롤링 실패(상품 수 급감, 빈 응답 등) 감지 시 Slack/이메일 알림 → 수동 확인 플로우를 파이프라인에 포함해야 한다.

11.3 크롤링 파이프라인

# Step 1: 목록 페이지 → 상품 URL 수집
def crawl_list_page(url):
    page.goto(url)
    text = page.inner_text('main')

# Step 2: 상품 상세 + 옵션 수집
def crawl_product_detail(url):
    page.goto(url)
    base_info = page.inner_text('body')
    page.click('[class*="option"]')
    page.wait_for_selector('[role="dialog"]')
    options = page.inner_text('[role="dialog"]')
    return parse_to_json(base_info, options)

# Step 3: DB 저장 (변경 감지 포함)
def save_with_diff(new_data):
    existing = db.query("SELECT * FROM bundle_catalog WHERE plan_name = ?", new_data.plan_name)
    if existing and existing.base_price != new_data.base_price:
        db.insert("bundle_price_history", {
            "bundle_id": existing.id,
            "old_price": existing.base_price,
            "new_price": new_data.base_price,
            "change_reason": "crawl_update"
        })
        notify_affected_users(existing.id)
    db.upsert("bundle_catalog", new_data)

# Step 4: 실패 감지 (상품 수 이상 감지)
def validate_crawl_result(new_count, expected_min):
    if new_count < expected_min * 0.7:  # 30% 이상 감소 시 알림
        send_alert(f"크롤링 이상 감지: {new_count}건 (기대 최소 {expected_min}건)")
        raise CrawlValidationError("봇 차단 또는 페이지 구조 변경 의심")

11.4 크롤링 실행 주기

주기 내용 목적
매일 새벽 3시 목록 페이지만 빠르게 확인 → DB와 가격 비교 → 변동 있으면 해당 상품만 상세 크롤링 트리거 → 영향받는 유저 알림 가격 변동 감지 (가벼움)

MVP 단계에서는 사용자 수동 입력으로 시작. 이후 단계에서 금융결제원 오픈뱅킹 API 연동 고려.

항목 내용

비용 무료
요건 사업자등록증, ISMS 또는 예외 인정
기간 심사 2~4개월
